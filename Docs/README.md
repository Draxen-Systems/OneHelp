## 📖 Sobre a Pasta

Esta pasta contém toda a base de **Engenharia de Software, Design Thinking e Regras de Negócio** do projeto OneHelp. 

Antes de qualquer linha de código ser escrita, o sistema foi devidamente planeado e documentado para garantir que resolve os problemas reais da ONG e dos seus utilizadores (funcionários, voluntários e adotantes).

## 📂 Estrutura de Ficheiros

Aqui encontra os documentos essenciais que orientam o desenvolvimento do projeto. A maioria dos ficheiros visuais está disponível tanto em formato `.pdf` quanto `.png` para facilitar a visualização direta no GitHub.

### 📄 Documentação Central
- **`Documentação OneHelp.docx`**: O documento principal. Contém o levantamento de requisitos (Funcionais e Não Funcionais), regras de negócio detalhadas e o âmbito geral do sistema.

### 🎨 Design Thinking e Experiência do Utilizador (UX)
- **`/persona`**: Perfil semi-fictício do nosso utilizador ideal (ex: adotante, doador ou voluntário), ajudando a guiar as decisões de interface e funcionalidades.
- **`/mapa_empatia`**: Mapeamento do que a Persona "pensa, sente, ouve e fala", para entendermos a fundo as suas dores e necessidades.
- **`/mapa_mental`**: Um fluxograma visual da ideação e das ramificações que compõem o ecossistema OneHelp.

### 💼 Estratégia de Negócio
- **`/business_model_canvas`**: O quadro de modelo de negócios. Define a nossa proposta de valor, parcerias principais (ex: clínicas veterinárias), estrutura de custos e canais de comunicação com o público.

### ⚙️ Modelação de Sistema (UML)
- **`/Diagramas`**:
  - `Diagrama_Caso_Uso_Simples.png`: Visão rápida das interações básicas entre os Atores e o Sistema.
  - `Diagrama_Caso_Uso_Completo.png`: Visão aprofundada de todas as funcionalidades e permissões de cada utilizador (Administrador, Funcionário, Adotante) dentro do sistema.

<details>
  <summary><strong>💡 Dica de Leitura: Por onde devo começar?</strong></summary>
  
  <br> 

  Se é um **novo programador** ou **membro da equipa** a analisar o projeto pela primeira vez, recomendamos a seguinte ordem de leitura para um melhor enquadramento:

  1. Leia o **Business Model Canvas** para entender o *propósito* geral e como o projeto se sustenta.
  2. Analise as **Personas** para saber para *quem* estamos a desenvolver a aplicação.
  3. Estude os **Diagramas de Caso de Uso** para ter uma visão clara de *tudo o que o sistema permite fazer*.
  4. Por fim, consulte a **Documentação OneHelp.docx** para esclarecer regras de negócio específicas (ex: "Quais os critérios para a triagem de uma adoção?").

</details>
